# RFC-169 · 资源页双栏化——左卡片列表 + 右详情编辑（agents / skills / mcps / plugins）

- 状态：Draft
- 日期：2026-07-11
- 发起：用户「代理的管理界面太丑了，是不是可以换成左侧 agent 卡片列表，右侧详情编辑的模式」
- 范围拍板（用户 4 问确认，2026-07-11）：
  - **D1 范围**：agents / skills / mcps / plugins 四个资源页一次改完（同一双栏骨架）；workflows 点开是画布、workgroups 点开是房间，两页不适用、维持现状。
  - **D2 新建形态**：右侧内联新建——点「+ 新建」后 URL 进 `/{res}/new`，左栏列表保持在场，右栏变空白新建表单；创建成功后自动选中新资源留在原地。
  - **D3 默认选中**：直接进入 `/{res}`（未选中）时右栏显示**空态引导**（「从左侧选择或新建」），不自动选中第一项。
  - **D4 左栏能力**：搜索过滤框 + 运行时/可见性等徽标 + 未保存修改指示（含切走前确认）三项全要。
  - **D5 右栏分组（用户追加，2026-07-11）**：「右侧编辑界面，要划分不同的配置，不要直接堆到一个长的框里往下滚动，太难看了」——右栏内部用**页签（TabBar，RFC-150 公共原语）**把配置切成组，每组一屏呈现；废除「一列 FormSection 堆到底」的长滚动形态（RFC-155 的折叠 section 语义由页签取代）。

## 1. 背景

四个资源管理页（`/agents`、`/skills`、`/mcps`、`/plugins`）目前是同一套「表格列表 → 点开整页表单」模式（RFC-151 已把 query/删除/owner 查询收敛进 `useResourceList`，把详情 header 收敛进 `DetailHeaderActions`）：

- 列表是 `data-table` 表格行，视觉密度低、行动按钮（打开/删除/启动…）挤在行尾；
- 点「打开」后整页跳转到 `/{res}/$name` 表单，**列表上下文完全丢失**——改完一个想看下一个，要保存→跳回列表→再点开，来回三跳；
- 保存成功后强制 `navigate` 回列表页，连续编辑体验割裂；
- 新建也是独立整页，同样丢上下文。

用户明确表示这套界面「太丑」，希望换成**左侧卡片列表 + 右侧详情编辑**的经典 master-detail 模式。仓内已有同构先例：RFC-072 任务输出双栏（`task-outputs-panel`，左端口列表+右详情）、RFC-168 工作组成员卡画廊+上下文面板。本 RFC 把这一模式确立为**资源管理页的标准骨架**并一次覆盖四页。

## 2. 目标

1. 四页统一换成双栏骨架：左栏 = 搜索框 + 资源卡片列表 + 「+ 新建」按钮；右栏 = 空态引导 / 编辑表单 / 内联新建表单，随 URL 切换。
2. 双栏骨架落成**公共组件**（新公共原语，遵守 Frontend UI consistency 原则），四页复用同一实现，卡片视觉完全一致。
3. 深链全部保留：`/{res}/$name`（plugins 为 `/$id`）与 `/{res}/new` 继续可直达（依赖树节点点击、工作流画布跳转、收藏夹等外部入口不受影响），选中态由 URL 驱动。
4. 连续编辑不丢上下文：保存后**留在原地**（不再跳回列表），列表卡片就地刷新；切换选中不重置左栏搜索词与滚动位置。
5. 未保存防丢：右栏表单有未保存改动时，对应卡片显示脏标记圆点；任何站内导航（点其他卡、点侧边栏）先弹确认对话框（公共 `Dialog`），浏览器关闭/刷新走 `beforeunload`。

## 3. 非目标

- 不动 workflows / workgroups 两页（画布/房间形态，不适用）。
- 不动 tasks / reviews / users / repos / clarify 等非 ACL 资源的 data-table 页（沿 RFC-151 D1 的伪抽象警戒——它们没有「选中一个改一个」的编辑语义）。
- **零 migration**；后端仅两处小件（均堵现存漏洞）：① 统一保护谓词让 `SKILL.md` 在写入**与删除**两入口都不可被路径别名（`./`、大小写、尾随分隔符、dot-segment）命中——堵现存漏洞（R2-P1-3/R3-P1-4/R4-P1-1）；② 探针 `startedAt` 捕获前移到配置快照读取之前——堵新鲜度 TOCTOU（R3-P2-5）。**skills 详情保存/读取沿用现状 double-PUT LWW 基线（无 CAS）；双栏前端只做「保存留原地 = 简单互斥（版本操作互不交叠）+ 成功后刷新版本历史并重播种草稿」**——不建精密版本一致性 gate（其深层洞是后端非原子双 PUT 症状、随版本权威一并转 170）；保存路径零后端改动、不比现状更差。详见 design §5.2/§5.3。
- **skills 存储层与全域并发一致性加固整体转 [RFC-170](../RFC-170-skills-storage-acl-hardening/proposal.md)**，不在本 RFC 范围：不透明复合 token（防同名删除重建 ABA）· 文件写删/ZIP/fusion 的全域乐观并发 · 版本快照权威存储模型 + 原子发布 + 崩溃恢复 · 运行时注入隔离全链 · 存量分叉待迁移决策 · 源生命周期权限校验 · 创建并发保留 · 权限接口版本化 CAS。这些都是**先于本 RFC 的现存缺陷**（今天就存在同名 ABA、fusion 越权、权限过期写回等）；169 的 skills 保存/读取沿用现状 double-PUT LWW，一个都不碰、不引入也不放大，整套版本锁/OCC/CAS 交 170 统一加固（详见 §6a 已知限制）。
- 不做移动端专属布局（本产品是桌面工具，与现有页面一致）；仅保证窄视口下双栏降级不破版（见 design）。
- 不引入自动保存；保存语义维持显式 Save 按钮（与全站一致）。
- 不改各表单字段本身的控件与校验（Field/TextInput/Select/JsonField 等原样复用）；改的是**分组呈现方式**——由堆叠折叠 section 变为页签（D5），RFC-155 的「折叠 + 有内容自动展开」语义随之退役，由页签徽标（内容计数）承接其「提示这里有东西」的意图。

## 4. 用户故事

1. 我打开 `/agents`，左边一列代理卡片（名称、描述、运行时/可见性徽标），右边提示「从左侧选择一个代理，或新建」。我点 `code-worker`，右边直接出现它的编辑表单；改完描述点保存，卡片上的描述就地更新，我**仍停在** `code-worker`，接着点下一张卡继续改。
2. 我在搜索框敲 `audit`，左栏立刻过滤出名称/描述命中的卡片；点开其中一张编辑，左栏的过滤词和滚动位置不变。
3. 我改了提示词还没保存，对应卡片出现脏标记圆点；此时我点了另一张卡，弹出「有未保存修改」确认框——选「留在本页」回去继续改，选「放弃修改」才切走。
4. 我点「+ 新建代理」，右栏出现空白新建表单（YAML 导入按钮还在），左栏列表仍可见；填完点创建，新代理出现在左栏并自动选中，右栏切到它的编辑表单。
5. 我从工作流画布点某个 agent 名跳到 `/agents/xxx`，看到的是双栏页面且 `xxx` 卡片高亮选中——深链行为与页内点选完全一致。
6. 我点开 `code-worker`，右栏不再是一条望不到底的长表单：五个页签一眼看清配置全貌，「提示词」页签里 Markdown 编辑器占满整个内容区高度，写长提示词不再挤在小框里；「资源与依赖」页签上的数字徽标提醒我这个代理挂了 3 个引用。

## 5. 交互规格

### 5.1 布局

```
┌────────────────┬──────────────────────────────────────┐
│ {资源} 标题     │ code-worker       [启动][ACL][保存][删除]│
│ 🔍 搜索…        │ ┌──────┬──────┬──────┬─────────┬─────┐│
│ ┌────────────┐ │ │ 基础 │提示词│ 端口 │资源与依赖│ 高级 ││ ← 页签固定
│ │code-worker ●│ │ └──────┴──────┴──────┴─────────┴─────┘│
│ │写代码 ·oc   │ │                                       │
│ └────────────┘ │   名称   [code-worker]                 │
│ ┌────────────┐ │   描述   [负责写代码的工人]             │
│ │auditor     │ │   运行时 [继承默认 ▾]                   │
│ └────────────┘ │                                       │
│                │   ← 当前页签内容独占一屏，              │
│ [+ 新建代理]    │     仅超高时页签内容区自身滚动          │
└────────────────┴──────────────────────────────────────┘
```

- 左右两栏**独立滚动**（沿 `.content` 视口钉死 + 页面变体 `:has()` 先例，见 design §布局）；左栏固定宽（约 260–300px），右栏弹性。
- 右栏纵向结构三段：**详情 header（固定，不随滚动消失）→ 页签条（固定）→ 当前页签内容（超高时仅此区滚动）**。
- 卡片：标题（资源名）+ 副标题（描述截断一行）+ 徽标行；整卡可点（`Link`），选中卡高亮（沿 `task-outputs-panel__option.is-selected` 的 accent 底色语言）；脏标记为标题旁小圆点。
- 卡片**零行内按钮**——所有行级操作迁入右栏（见 5.3）。

### 5.1.1 右栏页签划分（D5，各页配置组）

| 页 | 页签（依次） | 说明 |
| --- | --- | --- |
| agents | **基础**（名称/描述/运行时）· **提示词**（Markdown 编辑器占满内容区高度）· **端口**（输入/输出端口编辑器，徽标=端口数）· **资源与依赖**（技能/MCP/插件/依赖代理 pickers + 自动识别 + 依赖树预览，徽标=引用数）· **高级**（syncOutputsOnIterate/角色/端口重命名/权限/frontmatter 额外字段） | RFC-155 六个堆叠 section 重组为五页签；「折叠 + 有内容自动展开」由页签内容计数徽标取代 |
| skills | **概览**（描述 + 来源/路径等元信息）· **内容**（SKILL.md，external 只读）· **文件**（SkillFileTree——其内部文件列表+编辑器双栏在此页签独享整栏宽度，嵌套宽度问题自然化解）· **历史**（版本历史，仅 managed，能力门控沿 `skillCapabilities`） | Fuse 按钮维持 header extra |
| mcps | **配置**（McpFields）· **工具与探测**（McpInventoryPanel + 重新探测） | 探测面板从「表单上方堆叠」移入独立页签 |
| plugins | **配置**（PluginFields）· **更新**（当前版本/解析版本、检查更新、升级按钮与结果） | 原列表行的检查更新/升级操作落进「更新」页签 |

- 页签为右栏内局部状态（不进 URL，沿 settings 页先例）；切换资源时重置到首个页签。
- 新建视图同样用页签形态（agents 五页签同构；mcps/plugins 单「配置」组不显示页签条；skills 新建维持其自有的 managed/external/文件夹/ZIP 四模式页签）。
- 草稿跨页签共用一份，header 的保存按钮一次提交全部页签的改动；脏标记与页签无关。

### 5.2 URL 与右栏内容

| URL | 右栏 | 左栏选中 |
| --- | --- | --- |
| `/{res}` | 空态引导（EmptyState + 「+ 新建」CTA；skills 页额外承载源文件夹管理面板，见 5.4） | 无 |
| `/{res}/new` | 内联新建表单（原 new 页内容原样迁入，含 agents 的 YAML 导入、skills 的四 tab） | 无（「+ 新建」按钮呈激活态） |
| `/{res}/$name`（plugins `/$id`） | 编辑表单（原 detail 页内容原样迁入） | 对应卡片高亮 |

- 保存成功：**留在 `/{res}/$name`**，列表缓存失效就地刷新（行为变更：现状是跳回列表）。
- 创建成功：`navigate` 到新资源的 `/{res}/$name`（plugins 为 `/$id`），左栏出现并选中。
- 删除成功：`navigate` 回 `/{res}` 空态（现行为不变）。
- 资源不存在（深链失效）：右栏显示现有错误呈现（ErrorBanner 语言），左栏正常。

### 5.3 行级操作迁移（产品可见变更）

| 页 | 现状（表格行尾按钮） | 迁移后 |
| --- | --- | --- |
| agents | 启动 / 打开 / 删除 | 打开=点卡；启动、删除在右栏 header（已存在，`DetailHeaderActions` extra/del） |
| skills | 打开 / 删除 | 同上；Fuse 按钮维持右栏 header extra |
| mcps | 打开 / 删除 + **可展开行**（探针状态、工具 chips、重探针） | 打开=点卡；探针信息由卡片徽标（状态 chip）+ 右栏「工具与探测」页签（`McpInventoryPanel`，已含重探针）承接；**列表展开行体系退役**；status/latency/toolCount 三列信息保留在卡片徽标（状态）与探测页签（延迟/工具数） |
| plugins | 检查更新 / 升级 / 打开 / 删除 | 打开=点卡；**检查更新、升级迁入右栏「更新」页签**；卡片徽标显示「有可用更新」chip（检查过之后） |

### 5.4 各页卡片徽标与特有承接

- **agents**：运行时名（继承默认时显示「默认」中性 chip，沿现表格语义）、`private` 可见性、`builtin`。
- **skills**：`managed`/`external` 源类型 chip（沿现有 `chip--managed/--external` 配色）、`private`。列表页的 `SkillSourcesCard`（源文件夹 rescan/remove 面板）迁入**右栏空态视图**（未选中技能时显示：空态引导 + 源面板）——源管理是全局性操作，语义上属于「未聚焦某个技能」的状态。
- **mcps**：`local`/`remote` 类型 chip、enabled 关闭时「已禁用」chip、探针状态 chip（复用 `McpProbeStatusChip`）、`private`。
- **plugins**：`npm`/`file`/`git` 源 chip、版本号、有更新时更新 chip、enabled 关闭时「已禁用」chip、`private`。
- 徽标一律走 `StatusChip`/既有 chip class，不新造视觉。

### 5.5 未保存守卫

- 脏判定 = 右栏草稿与最近一次种子（加载/保存快照）的稳定序列化不等（纯函数，见 design）。
- 拦截面：站内一切路由跳转（页内点卡、点「+ 新建」、侧边栏、依赖树节点点击）统一走 router 级 blocker + 公共 `Dialog` 确认（「放弃修改 / 留在本页」）；浏览器刷新/关闭走 `beforeunload` 原生提示。
- 脏保护 = **父草稿级**（三次定案）：guard 拦截基于父草稿（主表单）dirty；子组件本地缓冲（无效 JSON、单个未存技能文件、ZIP 暂存）维持现状 best-effort——「保存留原地」使承载它们的组件保持挂载、比现状（导航离页即卸载）更安全。保存/创建成功后父草稿无残余脏时导航不弹确认。删除：确认冻结、请求在途编辑冻结、失败逐字恢复、成功回列表空态（矩阵㉛）。
- 现状四页均无任何防丢保护，此项为纯增益。

## 6. 验收标准

1. 四页均为双栏：左栏搜索框实时过滤（名称/描述，大小写不敏感）、卡片徽标符合 5.4、「+ 新建」在列表尾部；右栏三态（空态/编辑/新建）随 URL 正确切换。
1a. 右栏为「header 固定 + 页签条固定 + 页签内容区滚动」三段结构，页签划分符合 5.1.1；任何页签下都看不到跨组的长滚动表单；agents「提示词」页签的编辑器撑满内容区高度；端口/资源页签徽标计数正确；切换资源后页签复位到首个。
2. 深链 `/{res}/$name`、`/{res}/new` 直达时左栏在场且选中态正确；`/agents/$name` 的既有外部入口（依赖树点击、画布跳转、`ResourceNameCell`）不需要任何调用方改动。
3. 保存后停留原地且列表卡片就地刷新；创建后自动选中新资源；删除后回空态。
4. 脏标记圆点 + 切走确认 + `beforeunload` 全链可用；保存成功后立即导航不弹确认。
5. 行级操作迁移完成：mcps 展开行退役但探针信息不丢（卡片 chip + 「工具与探测」页签）；plugins 检查更新/升级在「更新」页签可用且卡片更新 chip 正确（与 5.1.1/5.3 同口径）；agents 启动按钮在详情 header（builtin 不显示，现状语义）。
6. skills 特有：源文件夹面板在空态视图可用（rescan/remove 全功能）；详情内嵌文件树双栏在右栏内不破版（≥1280px 视口无横向滚动）；四 tab 新建流程完整迁入右栏。
7. 全部现有测试迁移后 `bun run typecheck && bun run test && bun run format:check` + frontend vitest 全绿；`agents.png` 视觉基线刷新（其余三页不在基线集）；明暗双主题 + 窄视口视觉自查通过。
8. i18n：所有新文案（搜索 placeholder、空态引导、未保存确认、脏标记 aria）中英双语齐备，无硬编码字符串。

## 6a. 已知限制（诚实声明，交 RFC-170 加固）

169 的 skills 保存/读取**沿用现状 double-PUT LWW 基线**（无 CAS）；双栏前端只做「保存留原地」= 页面级 `operationBusy`（覆盖版本操作 + 后续刷新全程、期间全版本写与历史操作禁用）+ 成功后刷新版本历史并重播种草稿 + restore 在途冻结编辑器（§5.2；不建精密版本一致性 gate，深层一致性转 170）。下列**先于本 RFC 就存在**的一致性缺陷不在 169 范围（169 一个都不碰、不引入也不放大），交 RFC-170 统一根治：

- **子组件本地缓冲非完美追踪**：无效 JSON（无法保存）、单个未保存技能文件、ZIP 暂存不进父草稿 dirty、guard 不追踪——仅在用户主动导航离页时随卸载丢失（与现状同；stay-in-place 反使其更安全）。完美子缓冲零丢失（reporter registry/两阶段 discard）超出「不比现状差」，随后续 UI RFC 再评估。
- **skills 保存无版本锁（LWW）**：现状详情保存就是双 PUT last-write-wins、无任何乐观并发控制——169 不改它。**combined-save 单请求 + 单 fenced detail-read + contentVersion CAS 整套是 RFC-170 的入口**：slim 复审第四轮证明「基础 CAS 叠在现状非原子发布上不安全」（5xx 后重载读到「新版本号 × 残缺正文」，下次 CAS 反铸出更高版本永久回退内容），必须与 170 的快照权威 + rename 原子发布**同时**落地才安全，不可从 170 剥离。
- **同名删除重建 ABA / 文件写删/ZIP/fusion 全域一致性**：均无版本栅栏（现状），170 用 skillId 复合 token + 全域 OCC 根治。
- **快照权威存储模型**：live 目录是否唯一权威、原子发布、崩溃后半棵树恢复、运行时注入隔离——今天的 `commitSkillVersion` 就是「先删后拷」无原子性，170 引入快照权威 + rename 发布 + quarantine。
- **源生命周期权限**：源禁用/删除/重扫描按 registrar 身份删 skill 行，技能 owner 转移后旧 registrar 仍可删——现存越权，170 补逐 child ACL。
- **权限接口过期写回**：`PUT .../acl` 现无版本 CAS，过期请求可写回旧 owner 夺权——这是**全六类 ACL 资源的现存漏洞**，170 在共享服务加 `aclRevision`（届时所有资源受益）。

这些都不是 169 的双栏改造引入的；用户 2026-07-12 明确拍板拆分，让 UI 改造尽快落地、存储加固独立走 RFC-170 的设计门与实现。
