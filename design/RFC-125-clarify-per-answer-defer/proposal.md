# RFC-125 — 任务问题延迟下发改为默认开启（移除 launch 开关，统一延迟）

状态：Draft（待用户批准进入实现）
**演进**：本 RFC 初版想做"反问页 per-answer 逐条选延迟/立即"，经 **2 轮 Codex 设计 gate 共 6 条 finding** 揭示——`deferred_question_dispatch` 旧时**同时管 park gate 与反馈投递路径**，per-answer 的"同一任务混合立即+延迟"会触发逐条目误 park、级联重跑丢 provenance、在飞态 migration 缺口等一连串调度器高发区难题。用户据实测成本**拍板 Pivot：全部任务始终延迟**（放弃 per-answer 混合），换取简单与安全。

触发：2026-06-29 用户 RFC-124 收尾后问「批量下发/新增问题为何看不到」→ 厘清被 launch「是否延迟下发」开关门控 → 多轮设计反问后选定 Pivot。

## 1. 背景

RFC-120 T9（model A）给任务加了**「延迟下发」**：跨节点反问里**设计者域**答案，默认提交即**立即重跑**承接节点；若任务在 **launch 时**勾了 `deferredQuestionDispatch`，则改为**停进「问题看板」**待人手动「加入待下发→批量下发」。`新增问题`（手动问题）也只在延迟任务上出现。

问题：`批量下发`/`新增问题` 这套看板能力被一个 **launch 一锤子开关**藏起来——没勾的任务永远看不到（用户实测困惑）。而把"延迟决定"下沉为**每次答题逐条选**（per-answer 混合）被设计 gate 证实是**调度器高发区大改**（详见 §演进）。

## 2. 目标（Pivot）

- **G1**：**新任务一律默认延迟下发**——设计者域反问答案统一停进问题看板，由人「批量下发」。`deferred_question_dispatch` 对新任务默认 `true`。
- **G2**：**移除 launch 的「是否延迟下发」开关**（UI / 提交载荷 / API 入参的用户可见面）——不再有这个一锤子决定。
- **G3**：`新增问题`/`复制`/作者表单 在（默认延迟的）新任务看板**常驻可见**（受 `deferred` flag 门控、现为真）。`批量下发` 维持**既有 `stagedShown>0` 门控**——设计者答案现默认进看板 → 「加入待下发」≥1 张后 `批量下发` 出现（Codex P2：它不随 flag 自动显示，也不该）。**净效果**：二者从"非延迟任务上无从触达"变为**可达**（新增问题常驻；批量下发 staged 后即现），解掉用户「看不到按钮」。
- **G4 复用既有、零调度器改动**：完全复用 RFC-120 既有延迟模型（park gate / 队列投递 / 批量下发 / 不变量 / stuck 检测均按 `deferred_question_dispatch` 分支，**一字不改**）。新任务走 `true` 分支、老任务保持其原值——**任一任务在其生命周期内 flag 恒定、绝不混合**，故 6 条 finding 全不存在。

## 3. 非目标

- **不做 per-answer 逐条选 / 同一任务混合立即+延迟**（设计 gate 证实是调度器高发区大改，用户放弃）。因此**反问答题页不加第三个按钮、不变**。
- **不动 self-clarify**（恒立即，范围内本就不涉及）。
- **不翻转已存在的在飞任务**：老任务（`deferred_question_dispatch` 既有值，多为 false）**保持原行为**，不迁移、不中途改 flag（避免把"已立即跑过、dispatched_at=NULL 的设计者条目"误 park——这正是 gate P1#1 的坑，只在"中途改 flag"时出现，本 Pivot 通过"创建时定值、终生不变"规避）。
- **零 migration / 零调度器 / 零投递路径改动**。

## 4. 用户故事

- 作为任务发起人，我新建任务后打开「问题」标签页，`新增问题`/`批量下发` 就在那儿；设计者反问答案统一进看板，我攒齐后一次性批量下发——不必在 launch 时为整个任务做一锤子决定，也不会"当初没勾就永远看不到"。
- 作为维护者，我确认这是**复用既有延迟模型 + 改默认 + 删开关**，调度器/投递/下发执行**一字未改**，老任务行为不变。

## 5. 验收标准

1. 新建任务默认 `deferredQuestionDispatch=true`；launch 页**无**「是否延迟下发」开关。
2. 新任务「问题」看板：设计者反问答案进看板（待指派）；`新增问题`/`复制` 常驻可见；`批量下发` 在「加入待下发」≥1 张后出现（既有 staged 门控，不改）。
3. 设计者域反问答案**不再自动立即重跑**（统一手动批量下发）——这是本 RFC 有意的行为变更，需在 release note / STATE 标明。
4. **老任务零回归**：已存在任务 flag 值不变、行为延续；调度器/队列投递/批量下发/不变量/stuck 检测代码逐字未改。
5. self-clarify 反问、立即路径（老任务）逐字不变。
6. 测试：新任务默认延迟（launch 入参 / 创建默认）；看板按钮在默认延迟任务可见；`rfc120-deferred-dispatch` / `rfc120-manual-questions` 既有锁继续绿；`launch-deferred-dispatch` 收口（开关已删）。
7. 门槛：`typecheck + 前端 vitest + 后端 bun test + format` 全绿 + CI 全绿 + Codex 设计 gate（pivot）+ impl gate fold。
