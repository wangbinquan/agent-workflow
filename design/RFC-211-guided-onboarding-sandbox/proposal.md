# RFC-211 — 引导式沙盒：新用户上手引导与 example 产物生命周期

> **状态**：Draft（2026-07-20 起草，等待用户批准后实施）
>
> **触发**：用户 2026-07-20 提出「程序新装完之后能内置几个工作流、工作组、agent、skill 让使用者快速进入状态；每个新用户登录时也能看到这些向导元素；引导大家学会创建 agent、创建 skill、编辑工作流、启动工作流、编辑工作组、工作组 / 单 agent 等基本操作」。随后在设计问答中把方向收敛为：**引导过程中创建的元素默认为 example 形态，可在引导集合里一键清除这些 example 元素与任务；考虑多人同时跑引导，example 元素带 uuid 后缀防重名冲突。**
>
> **范围**：首页引导入口、引导 run/artifact 的服务端模型、六个核心动作的四条教程线、example 资源的创建与标记、一键清除（含任务产物）、现有 `Onboarding` 首跑卡片的重构与过期文案修正。

## 1. 现状盘点（15 路只读调研，均带源码锚点）

### 1.1 已经存在的东西

- **首跑引导卡片**：`packages/frontend/src/components/Onboarding.tsx` 提供四步走 + `PipelineHero` 动画 + `CapabilityGrid`（RFC-190），由 `/` 路由在首跑条件成立时整页渲染（`routes/index.tsx:26`）。
- **一个 demo 工作流**：`packages/frontend/src/fixtures/demo-workflow.ts`，三节点单 agent，作为前端常量存在，点按钮才走 `POST /api/workflows/import`。
- **两处播种先例**：`seedFusionResources`（RFC-101 内置 agent + workflow，repair-or-create 语义，`services/fusion.ts:207`）与 `seedBuiltinRuntimes`（空表守卫，删了不复活，`services/runtimeRegistry.ts:629`），都挂在 `cli/start.ts` 第 5e/5f 段、migrate 之后 HTTP 之前。
- **ACL 已经解决「新用户可见性」**：`visibility` 缺省 `'public'`，public 行对所有活跃用户可见（`services/resourceAcl.ts:93/176`）。这一层不需要任何新机制。

### 1.2 四个结构性缺口

1. **现有首跑判据是实例级、且与「装完就有示例」互斥。** `computeIsFirstRun` 要求 agents 与 workflows **两个列表都为空**（`Onboarding.tsx:34-42`）。这带来两个后果：**（a）** 只要实例里存在任意可见资源，第二个登录的用户永远看不到引导——「每个新用户登录时也能看到」在现状下**不成立**；**（b）** 任何形式的开机播种都会让 `isFirstRun` 恒为 `false`，**内置示例反而会把引导永久关掉**。
2. **`builtin` 标记不能承载「示例」语义。** RFC-104 把它钉死为「列表隐藏（`excludeBuiltinAgents/Workflows`，`systemResources.ts:54/60`）+ 全面只读（`assertNotBuiltin`，`systemResources.ts:75`）+ 禁止启动（`taskLaunchGate.ts:34`）」，与「看得见、跑得动、改着学」三项要求逐条相反。且 `skills` / `workgroups` 两张表**连这个列都没有**。
3. **没有任何 per-user 状态载体。** users 表无偏好列、无 KV 表、`/api/auth/me` 不返回可判「新用户」的字段、前端 localStorage 也没有对应键。`last_login_at` 不可用作首登判据（OIDC 登录路径根本不写它，`routes/auth.ts:51` 只在密码登录写）。
4. **现有 demo 是坏的、文案是过期的。** demo 引用的 `coder` agent 从不存在，导入后启动必 `agent-not-found`（fixture 注释自认）；引导第 4 步文案还在教「选一个本地 git 仓 + base 分支」，而 RFC-165 早已删除本地路径模式，只剩 remote URL ⊕ scratch。

### 1.3 三个连带发现（本 RFC 顺手处理或显式记为非目标）

- **「从模板开始」在全新装机上恒不可用**：workflow starters 需要若干带特定 role / 端口的 agent，空库下这条产品自带的最强教学入口被锁死。本 RFC 的引导产物天然解锁它（非目标，但会在引导文案里指向它）。
- **未装 opencode 时 daemon 直接 `exit(1)`**（`cli/start.ts:100-133`），用户连 UI 都进不去，任何前端引导对「真正的第一次」无效。这是产品级 fail-closed 策略变更，**列为非目标**，但引导在启动任务前会做运行时就绪度探测并给出可读提示。
- **新手最容易踩的坑无护栏**：`outputs` 为空的 agent、空 `promptTemplate` 的节点都能保存且能启动，跑出来「什么都没发生」。本 RFC 用「帮我建」生成的样板绕开它，**不改 validator**（列为非目标，记为后续输入）。

## 2. 目标

1. **新用户第一次登录就看得见一个入口**，判据是「这个用户」而不是「这个实例」，且完成/跳过之后仍可随时重开。
2. **六个核心动作都能在引导里真正做一遍**：创建代理、创建技能、编辑工作流、启动工作流、编辑工作组、跑一次（工作组或单 agent）。
3. **每一步都同时支持「帮我建」和「我自己来」**：想快的人一键得到一个能跑的样板并直接进编辑页；想学的人深链到空白表单亲手填，建完引导自动打勾。
4. **引导产物是可回收的沙盒**：引导期间产生的一切资源与任务打 example 标记，可在引导集合里一键清除，**连带清理任务产物**（worktree / scratch 目录 / 运行日志）。
5. **多人可同时跑引导互不干扰**：产物归属创建者、默认 private、名字带同一次 run 的短 uuid 后缀，列表互不污染、命名不冲突。
6. **引导能带用户真跑通一个任务**，走 scratch 临时空间，不要求先接仓库（普通用户没有 `repos:write`）。

## 3. 非目标

- **不做开机播种**。新装完 DB 仍为空；所有 example 元素都在用户点「开始引导」之后才产生。这一条是用户明确拍板（详见 §5 D1），它同时消解了 §1.2 的缺口 1 与 2 的大半冲突面。
- **不推翻 RFC-199 的「零指导」立场**。不新建 tour / coachmark / spotlight 原语，不做遮罩式分步高亮；引导是**可见的界面 + 可读的样板 + 能打勾的清单**，不是播放式教程。
- **不做示例资源的「转正」动作**（去 example 标记、去 uuid 后缀、转 public）。列为扩展点；本版通过「清除是显式动作 + 二次确认列出清单」来防误删。
- **不覆盖高阶能力**：MCP、插件、记忆、定时任务、人在环评审、多进程分片不进首版主题，但扇形结构为它们留好扩展位。
- **不改 agent / 工作流的校验规则**（不给「零输出端口」「空提示词」加 warning），不改 `builtin` 语义，不改 `ResourceSplitPage` 之外的共享外壳。
- **不做未装 opencode 时的降级启动**。
- **不改「语言是全局配置」**这一现状。

## 4. 用户故事

**US-1（新装机的第一个用户）** 我装完启动，登录后首页上方有一张「上手引导」卡片，写着平台能干什么、以及四条可选的教程线。我点「造一个能干活的代理」，引导问我要「帮我建」还是「我自己来」；我选「帮我建」，两秒后我停在一个名叫 `guide-coder-7f3a2b` 的代理编辑页上，端口、提示词、运行时都填好了，旁边一行小字解释每个字段是干什么的。我改了两句提示词，回到引导，第一步已经打勾。

**US-2（想亲手学的用户）** 同一步我选「我自己来」，引导把我送到空白的新建代理表单，并在页面上留一条提示条说明「这是引导的第 1 步，建完会自动回到引导」。我填完保存，回到引导页，第一步自动变成已完成，并且我刚建的这个代理**也被标成了引导产物**（引导明确告诉我：清除引导时它会被一起删掉）。

**US-3（跑通第一个任务）** 教程线最后一步是「跑一次」。引导先检查运行时是否就绪；就绪就用 scratch 临时空间启动一个极小任务，我能实时看到节点在跑、跑完看到产出与 diff。不就绪就直接告诉我缺什么、去哪配，而不是让我看一个语焉不详的 spawn 失败。

**US-4（收拾干净）** 我学完了，回到引导集合页，点「清除引导产物」。弹窗列出将被删除的 3 个代理、1 个技能、2 个工作流、1 个工作组、4 个任务，我确认后它们连同 scratch 目录和运行日志一起消失，正在跑的引导任务先被安全取消。我自己后来建的正式资源一个没动。

**US-5（团队里同时上手）** 我们五个人同一天登录同一个实例各跑各的引导。我在 `/agents` 列表里只看得到自己的 `guide-coder-7f3a2b`，看不到同事的 `guide-coder-9c1d4e`；名字不冲突，列表不刷屏，谁清除只影响谁。

**US-6（老用户回来复习）** 我三个月前跳过了引导。今天想学工作组，首页那行小入口还在，点进去直接选「让一组代理协作」这条线，前面的代理我已经有了，引导检测到缺前置产物时会问我「用现有的还是帮你建一个新的」。

## 5. 产品决策

### D1 — 不播种，一切由引导创建（用户拍板）

新装完数据库仍为空。「快速进入状态」由**用户主动点一下引导**来达成，而不是开箱就塞一堆共享样板。

**理由**：① 播种会把首跑引导永久关掉（§1.2-1）；② 共享样板在多人实例里会被一个人改坏、影响所有人；③ owner 归属（`__system__` vs 首个 admin）没有干净答案；④ 播种会让每条 e2e 的临时 home 都带上示例，冲击 `fixture:'clean'` 语义与视觉基线；⑤ 引导创建路径天然产出**归属明确、可清除、可重来**的产物。

### D2 — 扇形结构：一个入口 + 自选主题（用户拍板）

首页一个入口，进去用 `ChoiceCards` 选主题。首版四条教程线覆盖用户点名的六个核心动作：

| 线 | 覆盖动作 | 步骤 |
| --- | --- | --- |
| **T1 造一个能干活的代理** | 创建 agent、单 agent 直跑 | ① 建代理 ② 认识输出端口 ③ 用它跑一次（scratch，真跑） |
| **T2 给代理装一个技能** | 创建 skill | ① 建技能 ② 挂到代理上 ③（可选）再跑一次看差别 |
| **T3 把代理串成流水线** | 编辑工作流、启动工作流 | ① 得到一个工作流 ② 在画布上连一条边 ③ 校验 ④ 启动（真跑） |
| **T4 让一组代理协作** | 编辑工作组、跑工作组 | ① 建工作组并选模式 ② 配成员与 leader ③ 启动（真跑） |

**主题之间有前置依赖**（T2/T3/T4 都需要至少一个代理）。规则：进入一条线时若前置产物缺失，引导**不报错**，而是给两个选择——「用我已有的 X」或「帮我补一个」。选前者时该资源**不打 example 标记**（用户的正式资源不因为被引导引用就变成可清除产物）。

高阶主题（MCP / 插件 / 记忆 / 定时任务 / 人在环评审 / 多进程分片）在同一张选择卡上留位，首版显示为「即将推出」或直接不渲染，由 §7 扩展点决定。

### D3 — 每步双路径：「帮我建」/「我自己来」（用户拍板）

- **帮我建**：调后端「引导产物创建」通道，一次生成该步骤的完整样板，然后**跳到这个资源的编辑页**（不是详情页）——用户第一眼看到的是可改的真实表单，而不是只读展示。
- **我自己来**：深链到对应的空白新建表单，URL 带引导上下文（run id + step key）。表单页顶部渲染一条引导提示条（复用 `NoticeBanner`），说明「你在引导第 N 步，保存后会自动回到引导」。**保存时把引导上下文一并提交给后端**，由服务端登记为该 run 的产物——不靠前端轮询列表猜「用户是不是建好了」。
- 两条路径产出的资源**同样**打 example 标记、同样进一键清除范围。引导 UI 必须**显式告知**这一点（US-2），这是本设计里最需要防误伤的地方。

### D4 — example 产物的身份、命名与可见性

- **归属**：`owner_user_id` = 当前用户。
- **可见性**：`private`（用户拍板）。多人同时跑引导时互不污染列表。
- **命名后缀**：**每次 run 一个共享短后缀**，取该 run ULID 的尾 6 位小写（如 `7f3a2b`），同一次引导产出的所有资源共享它 —— 一眼看出成组、便于人工识别与清理，也比「每个资源各自 uuid」可读得多。名字形如 `guide-coder-7f3a2b`、`guide-audit-flow-7f3a2b`。（后缀字符集与各资源 name 正则的兼容性由 design.md 逐条核实。）
- **标记**：**两处都写**（用户拍板）——`onboarding_artifacts` 表记录 run → 资源的归属关系（清除时按批次扫一张表），业务表上的 `example` 布尔列供列表 DTO 直接渲染「示例」角标与过滤。两处真相必须有一致性测试锁死。

### D5 — 一键清除的范围与安全性（用户拍板：只清自己的 + admin 全局入口 + 连产物一起删）

- 普通用户：`POST /api/onboarding/cleanup` 清除**自己**的全部 example 资源与任务。
- 管理员：额外的全实例入口（带二次确认，明确告知会影响其他用户正在进行的引导）。
- **连带清理任务产物**：worktree、scratch 目录、`runs/` 运行日志一并回收（「恢复出厂」语义）。
- **在跑的任务先安全取消**再删。
- **删除顺序按引用逆序**：任务 → 工作组 → 工作流 → 代理 → 技能，避开 `agent-in-use` / `skill-in-use` 之类的 409 守卫。
- **跨 DB + 文件系统不可能原子**：清除返回逐项结果，部分失败可重试且重试幂等；绝不因为一个目录删不掉就把整批回滚成「什么都没删」的中间态。
- **永不自动触发**：清除只在用户显式点击并通过列出清单的二次确认后执行。

### D6 — 真跑一次任务，走 scratch（用户拍板）

- 引导里的「跑一次」使用 scratch 临时空间，不要求先接仓库（普通用户没有 `repos:write`，接仓库这一步会直接断链）。
- **启动前探测运行时就绪度**，未就绪时给出可读的下一步（缺二进制 / 未登录 / 缺凭据），而不是让任务以节点级 spawn 失败收场。
- 引导任务是**真实**的 opencode 进程与真实 token 消耗；引导 UI 必须在启动按钮旁说明这一点。
- 任务规模刻意压到最小（单节点、短提示词、短超时），把首次体验的成本与等待都控制住。

### D7 — 入口位置与「每个新用户都能看到」

- **首页常驻**：`/` 顶部一张引导卡片。**展开态**判据 = 当前用户没有任何已完成的引导 run（per-user，不是 per-instance）；完成或显式收起后变成一行小入口，随时可重开。
- 现有整页 `Onboarding` 组件**重构**为这套入口的一部分：保留 `PipelineHero` 与 `CapabilityGrid`（它们仍是最好的「这平台是干什么的」表达），**替换**四步硬编码列表与那个跑不通的 demo 导入按钮，**修正**已过期的文案（step4 的「本地 git 仓 + 分支」）。
- 引导集合页（`/onboarding` 或等价路由）承载：主题选择、当前 run 的步骤进度、产物清单、一键清除。

## 6. 验收标准

1. 全新安装、第一个用户登录，首页出现展开的引导卡片；他跑完任一条线后卡片收起为小入口且可重开。**第二个用户**登录时，即便实例里已有大量资源，他仍看到展开的引导卡片。
2. 四条教程线的每一步都能用「帮我建」在一次点击内产出资源并落到该资源的编辑页；也能用「我自己来」深链到空白表单，保存后回到引导且该步自动打勾。
3. 「帮我建」产出的代理 / 工作流 / 工作组 / 技能全部**能真正跑通**：工作流过 `validateWorkflowDefinition` 且 `ok:true`，工作组过 launch readiness，代理有非空 outputs 端口。CI 有测试锁死这一条（不是「导入成功」，是「能启动」）。
4. 引导产物全部 `owner=创建者`、`visibility=private`、名字带同一 run 的短后缀；两个用户同时跑同名教程线不发生任何唯一约束冲突，且各自列表里看不到对方的产物。
5. 一键清除后：该用户的 example 资源、example 任务、对应的 worktree / scratch 目录 / 运行日志全部消失；**非 example 的资源一个都没动**；重复点击清除是幂等的；清除前的二次确认列出了将删清单。
6. 清除时若有引导任务在跑，它被先取消再删，不留孤儿进程与孤儿 worktree。
7. `onboarding_artifacts` 表与业务表 `example` 列两处真相一致，有测试锁死漂移路径。
8. 引导启动任务前会探测运行时；未就绪时给出可读提示而非失败的任务。
9. `bun run typecheck && bun run lint && bun run test && bun run format:check` 全绿；e2e 与视觉基线的冲击面在 plan.md 里逐条登记并处理完毕。

## 7. 扩展点（本版留位、不实现）

- 高阶教程主题：MCP、插件、记忆、定时任务、人在环评审、多进程分片。
- **产物转正**：把某个 example 资源「移出引导集合」（去标记、去后缀、可选转 public），使其不再受一键清除影响。
- 引导产物的分享/导出（例如把学完的工作流导出 YAML 给同事）。
- 未装 opencode 时的降级启动 + 产品内安装引导。
- agent / 节点的「有效性」软护栏（零输出端口、空提示词的 warning）。

## 8. 与既有 RFC 的关系

- **RFC-199 / RFC-201（零指导 UX）**：本 RFC **不推翻**其立场。RFC-199 的「零指导」针对的是**界面自解释**，本 RFC 提供的是**可回收的练习场**——两者互补。本 RFC 不引入遮罩式教程与 tour 原语。
- **RFC-104（builtin 语义）**：不改动。example 是与 builtin **正交**的新维度：builtin = 框架基础设施（隐藏 + 只读 + 禁启动），example = 用户自己的引导沙盒产物（可见 + 可改 + 可启动 + 可清除）。
- **RFC-099（资源 ACL）**：完全复用。引导产物就是普通的 private 资源，不引入任何新的可见性判定路径。
- **RFC-190（首页 hero / CapabilityGrid）**：保留并复用，替换其下的四步硬编码列表。
