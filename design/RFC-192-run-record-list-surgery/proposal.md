# RFC-192 · 运行记录列手术——任务 / 定时任务表格增强与跨页一致性收尾

- 状态：Draft
- 日期：2026-07-15
- 发起：同 [RFC-191](../RFC-191-resource-list-gallery/proposal.md)（2026-07-15 列表页形态调研）；用户确认决策点 D2（任务过滤纯前端）、D3（「立即运行」轻确认）、D4（相对时间口径）、D5（repos/users 一致性同批收尾）。
- 姊妹 RFC：RFC-191（定义资源画廊）。**依赖 RFC-191-T1**（`<RelativeTime>` 公共原语），实现顺序 191 → 192。

## 1. 背景

任务 / 定时任务是**运行记录**：状态是第一公民、条目随时间增长（任务列表 API 上限 500 行）。表格仍是正确载体——问题不在形态，在列结构与行内能力。调研实测：

**任务列表**（名称 · 工作流 · 状态 · 开始 · 仓库 · 错误 · 操作）：
1. 「错误」列常驻但绝大多数行是「—」——为少数失败行让所有行付一列宽；
2. 仓库列全路径 `<code>` 展示，前缀高度重复（`/Users/…/proj/`），真正的辨识信息（仓名）淹没在尾部；
3. 无耗时列（`finishedAt` 已返回）；运行中的任务无任何「活动感」；
4. `repoCount` 字段就是为列表「N repos」chip 而加的（schema 注释原话），但 UI 从未接线；
5. 行不可整行点击（定时任务页可以），行尾「打开」按钮冗余；
6. 过滤仅状态一维：不能按执行主体过滤、不能搜名字——这恰是唯一会增长到数百行的列表；
7. `scheduledTaskId` 已在列表 payload，定时启动的任务无溯源标记。

**定时任务列表**（名称 · 周期 · 下次触发 · 最近触发 · 启用）：
1. 「启用」是纯文本——启停这个页面最高频的操作要进详情页才能做，而 `PUT /api/scheduled-tasks/:id`（部分更新 `{enabled}`）就是详情页在用的端点；
2. 「最近触发」只有结果 chip：不显示 `lastRunAt` 时间、不链接 `lastTaskId` 跑出来的任务；
3. `consecutiveFailures` 不显示——连挂 N 次的调度与偶发失败无法区分；
4. 无「立即运行」入口，而 `POST /api/scheduled-tasks/:id/run-now` API 与详情页按钮均已存在；
5. 下次触发是绝对时间，不如相对时间直读。

**跨页一致性**（调研清单，五维）：行点击口径、冗余「打开」、删除确认两套、时间格式三套、ID 展示三套。其中前两项与时间格式由本 RFC 在任务 / 定时任务 / repos 三页收口（工作流 / 工作组由 RFC-191 收口）。

## 2. 目标

### 2.1 任务列表（运行监控形态）

1. **列重排**：`状态 | 任务 | 主体 | 仓库 | 开始 | 耗时 | ›`。
   - 状态列前置（监控扫读入口）；**运行中行状态 chip 带脉冲点**（`prefers-reduced-motion` 静止）；
   - 「任务」单元格 = 名称（Link）+ ULID 副标题（现状保留）+ **失败行第二行红色错误摘要**（判据 `status === 'failed' && errorSummary != null`——canceled / interrupted 行的 `errorSummary` 是「canceled by user」类说明文本，不按红色错误渲染；单行截断、title 全文）——「错误」常驻列退役；`scheduledTaskId` 非空时名称旁「定时」chip 链回 `/scheduled/$id`；
   - 仓库列只显示仓名（路径模式 = `repoPath` basename；URL 模式 `repoPath` 是缓存目录 `{hash}-{slug}`，改从**脱敏后的 `repoUrl`** 派生仓名，避免把内部缓存名当仓名展示）、完整信息进 title；`repoCount > 1` 补「N 仓库」chip；
   - 开始列 `<RelativeTime>`；新增**耗时列**（终态 = `finishedAt - startedAt`；运行中 = 「进行中 · X」活动时长；等待态 = 「等待 X」）。
2. **整行可点**进任务详情（定时任务页既有模式推广），行尾「打开」按钮退役、以 `›` chevron 提示可点。
3. **过滤三维化（纯前端，决策 D2）**：状态 chips（现状 URL 驱动，保留）+ 主体 `Segmented`（全部 / 工作流 / 工作组 / 单代理，`taskExecutionKind` 共享派生点）+ 名称搜索框。列表查询**显式传 `limit=500`**（`listTasks` 默认只回 100 行、路由上限 500——不传则本地过滤会静默漏掉第 101 条以后的匹配；Codex 设计门 P1）；过滤覆盖面即「最近 500 条」，超出部分需要分页/服务端过滤时另立 RFC（非目标注记）。零后端改动。

### 2.2 定时任务列表（行内操作）

4. **列重构**：`启用(Switch) | 名称 | 周期 | 下次触发 | 最近触发 | 操作`。
   - 启用列换公共 `Switch` 就地 `PUT {enabled}`（行点击不受影响，Switch 点击 stopPropagation）；
   - 下次触发 = 相对时间主行（「4 小时后」）+ 短格式绝对时间副行；禁用行「—」；
   - 最近触发三合一 = 结果 chip + `<RelativeTime ts={lastRunAt}>` + 任务链接（**仅 `lastStatus === 'launched'` 时渲染**——失败触发不产生任务，后端此时不更新 `lastTaskId`，无条件渲染会把失败 chip 链到一个更早的成功任务上；Codex 设计门 P1）；`consecutiveFailures > 1` 追加「连挂 ×N」danger chip；
   - 操作列 = 「立即运行」（**轻确认**：`ConfirmButton` 两击，决策 D3；成功后跳新任务，与详情页行为一致；「需修复」行禁用）。

### 2.3 一致性收尾（决策 D5）

5. repos 页 `lastFetchedAt` 改 `<RelativeTime>`（该页删除确认已是 `Dialog`、users 页无时间列且禁用流已有确认——盘点后收尾范围即此一项）。
6. 全站时间口径记档：**列表层相对时间 + title 绝对；详情层保持绝对时间**（tasks/scheduled 详情页不动）。

## 3. 非目标

- **零后端改动**：不加分页 / 服务端排序 / 新 query 参数（D2 拍板纯前端；未来若需分页另立 RFC）；不动任务状态机与生命周期。
- 不做时间分组页眉（今天 / 昨天）——先把列与过滤做对，分组属锦上添花。
- 不动 `tasks.detail` / `scheduled.$id` 两详情页（除其引用的公共组件自然受益外）；详情页绝对时间保留（§2.3 口径）。
- 不动 reviews / clarify / memory / users 列表结构（users 经盘点无需收尾项）。
- 主体过滤与搜索**不进 URL**（本地 state；状态过滤维持现状 URL 驱动不变）。
- 不做批量操作（多选取消等）。

## 4. 用户故事

1. 列表里三十个任务，我点「运行中」chip 再在搜索框敲 `RFC`，两秒定位到目标任务；它的状态 chip 带着脉冲点、耗时列写着「进行中 · 12 分」。我点行任意处进详情。
2. 一个失败任务的错误摘要就在名称下方一行红字，我不用横向扫到最后一列；旁边「3 仓库」chip 提醒我这是多仓任务。
3. 一个由夜间调度启动的任务名称旁有「定时」chip，点它直接跳到那条定时任务的详情。
4. `/scheduled` 页我直接拨 Switch 关掉一条调度；另一条「连挂 ×3」的红 chip 扎眼，我点它「最近触发」里的任务链接看日志，修好后点「立即运行」（再点一次确认）立刻补跑，页面跳到新任务。
5. repos 页「上次拉取」显示「3 小时前」，悬停能看到精确时刻——与任务、定时任务页同一套时间语言。

## 5. 交互规格

### 5.1 任务表

| 列 | 内容 | 备注 |
| --- | --- | --- |
| 状态 | `TaskStatusChip`；running 加脉冲点 | chip 语义色沿 `TASK_STATUS_KIND` |
| 任务 | 名称 Link + `<code>` ULID 副行；失败行错误红字第二行；「定时」chip（条件） | 错误行单行截断 + title 全文 |
| 主体 | `TaskSubjectLink`（现组件，badge 保留） | 不变 |
| 仓库 | `basename(repoPath)` + title 全路径；`repoCount>1` → 「N 仓库」chip | `<code>` 字体保留 |
| 开始 | `<RelativeTime ts={startedAt}>` | |
| 耗时 | 见 §2.1；pending → 「—」 | 文案前缀区分 进行中/等待 |
| › | chevron，无按钮 | 整行可点 |

工具条：状态 chips（现状）+ 右侧 `Segmented`（主体）+ 搜索框；**列表为空且无过滤词时新控件不渲染**（e2e `tasks.png` 空态基线零 churn）。

### 5.2 定时任务表

| 列 | 内容 | 备注 |
| --- | --- | --- |
| 启用 | `Switch`（无文字，aria-label） | `PUT {enabled}`；pending 禁用 |
| 名称 | 名称 + 「需修复」badge（现状判据不变） | |
| 周期 | `scheduleSummary`（现状） | |
| 下次触发 | 相对 + 副行短绝对；禁用/无值 → 「—」 | 未来向 relTime |
| 最近触发 | 结果 chip + 相对时间 + 任务链接；「连挂 ×N」（N>1） | 从未触发 → muted「未触发」 |
| 操作 | 「立即运行」ConfirmButton（sm） | 需修复行禁用；成功跳新任务 |

行点击进详情（现状保留）；Switch / 链接 / 立即运行 stopPropagation。

## 6. 验收标准

1. 任务表七列如 §5.1，错误常驻列消失、失败行错误可见可悬停；运行中行有脉冲、耗时列各状态文案正确。
2. 状态 × 主体 × 搜索三维过滤可组合，纯前端、互相 AND；空 DB 时页面与改版前字节一致。
3. 任务/定时任务两表整行可点、无「打开」按钮；行内交互元素点击不触发行跳转。
4. 定时任务表 Switch 就地启停、「立即运行」两击确认后跳新任务、需修复行禁用、连挂 chip 与任务链接可用。
5. repos 页时间列相对化；详情页时间保持绝对。
6. 五门（typecheck / lint / test / format / e2e）全绿，`tasks.png` 基线零 churn。
