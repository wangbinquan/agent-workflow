# RFC-206 · 焦点环裁剪问题的一次性根治与杜绝

> 产品视角。技术设计见 [design.md](./design.md)，任务分解见 [plan.md](./plan.md)。

## 1. 背景

用户 2026-07-20 报告：`/repos → 批量导入` 弹窗里选中输入框时**上边缘被切掉**，`/agents → 高级` 页签里的输入框**左右边缘被切掉**，并指出「还有好几个地方都是这样的」「为什么总是出这种问题，能不能一次性全部解决并杜绝以后再出现」。

这不是新 bug，是**第五次**复发。仓库里已经有四处针对同一现象的补丁：

| 补丁位置                                | 覆盖的轴             | 留下的锁                                       |
| --------------------------------------- | -------------------- | ---------------------------------------------- |
| `.dialog__body`（负 margin 借位）       | 仅左右               | `dialog-body-focus-outline-clip.test.ts`       |
| `.fuse-dialog .dialog__body`（RFC-101） | 仅左右（与上条重复） | —                                              |
| `.task-detail__pane > .workgroup-room`  | 四边                 | `workgroup-room-composer-outline-clip.test.ts` |
| `.page--editor > .form-grid`            | 仅左右，且只有 2px   | `editor-layout-viewport-fit.test.ts`           |

每一次都只补了「这次被报出来的那一条轴」，所以每次都会从**没补到的轴**再冒出来。用户这次报的两处，正是四次补丁都没覆盖到的轴。

## 2. 为什么会反复复发（问题的问题）

根因不是某个人写错了 CSS，而是**两个互相独立、都在持续增长的集合，其交叉处没有归属**：

- **集合 A** —— 把焦点指示画在 border box **外面**的规则（`outline` 非负 offset、非 `inset` 的 spread `box-shadow`）。当前约 **36 条**。
- **集合 B** —— `overflow != visible` 的裁剪容器。当前 **100+ 条**。

bug 发生在 A×B 的笛卡尔积上：任意一个 A 元素恰好贴着任意一个 B 元素的裁剪边。四个结构性因素让它必然反复出现：

1. **写代码时不可见** —— 在容器上加 `overflow: auto` 的人，无法预见自己切掉了三层之下某个输入框的焦点环；反过来加一个满宽输入框的人，也不会去数它上面有几层滚动容器。
2. **测试在物理上测不到** —— jsdom **没有布局引擎**。现存所有相关测试都是对 `styles.css` 的**源码文本断言**，它们只能锁住「某条补丁的字面量还在」，**永远无法**发现「实际被切了」。
3. **review 看不见** —— A 与 B 的规则在同一个 17k 行样式表里相隔上万行。
4. **最关键：第一次的修法选错了复杂度** —— 首个补丁修的是**容器**。这教会了后续每一个人「出现这问题就给容器补 padding」。这是 **O(容器数)** 且随时间无限增长的工作量；而正确的修法是改**环**本身，只有 **O(1)**。

单纯再修一遍现存问题不解决任何事——第六次复发只是时间问题。**必须同时改变「修法」和「可见性」。**

## 3. 目标

1. **G1 · 让问题在 CI 里可见** —— 建立能在**真实布局引擎**里判定「焦点环是否被裁」的自动化守卫，把一个此前只能靠用户肉眼发现的缺陷，变成 CI 的确定性失败。
2. **G2 · 止血** —— 守卫上线即生效：任何**新增**的违规立刻被拦下，不再有新的复发。
3. **G3 · 清存量** —— 守卫输出的完整违规清单作为基线白名单，逐条修复，白名单最终清空。
4. **G4 · 转硬失败** —— 白名单清空后取消豁免机制，此后任何违规一律红。
5. **G5 · 修法归位** —— 把「补容器」的默认修法改成「环画在控件内部」，并把这条规则固化成可执行的守卫，而不是文档里的口头约定。

## 4. 非目标

- **不做全站焦点环视觉重做**。实心按钮（`.btn--primary`）保持现有外扩环。把 accent 环画进 accent 底色的按钮里会让它几乎不可见，需要重新设计双层环——那是独立的视觉议题，本 RFC 不碰（见 §7 备选方案 B）。
- **不重构滚动容器布局**。只在必要处增加内边距，不调整任何布局结构。
- **不处理 `outline: none` 且无替代指示的 a11y 缺失**（`.reviews-row__expand`、`.reviews-version-list__open`、`.repo-source-tabs__tab`）。那是「根本没有焦点环」，与「有环但被切」是不同缺陷；本 RFC 只在清单里登记，修复另立工单。
- **不覆盖 webkit**。几何审计依赖 Chrome DevTools Protocol 强制伪类（见 design.md §3.2），只在 chromium project 运行；webkit 夜间跑自动跳过。

## 5. 用户故事

- **作为使用者**，我在任何页面、任何弹窗里聚焦任何输入框、下拉、开关、按钮时，焦点指示都**完整可见四条边**，不会被容器切掉——无论是键盘 Tab 还是点击。
- **作为使用者**，我不需要再为同一类视觉缺陷反复提 issue。
- **作为开发者**，当我新增一个 `overflow: auto` 容器、或新增一个焦点样式时，**如果它切掉了任何控件的焦点环，CI 会直接失败**，并明确告诉我：哪个元素、被哪个容器、切了哪条边、差多少 px。
- **作为开发者**，我不需要记住「焦点环会被 overflow 裁剪」这条冷知识——守卫会替我记住。
- **作为 reviewer**，我不需要在 review 里靠肉眼推演 A×B 交叉。

## 6. 验收标准

1. **AC1** —— 新增 Playwright 几何审计 spec，遍历全部主要路由 + 主要弹窗，对每个可聚焦元素强制 `:focus`/`:focus-visible` 后测量实际几何；任一处 `room < ink` 即失败，错误信息包含元素选择器、裁剪容器选择器、被切的边、差值 px。
2. **AC2** —— 审计**能证伪**：在基线白名单为空的前提下，把 `.form-input:focus` 的 `outline-offset` 改回 `0`，审计必须变红（变异测试证明非空跑）。
3. **AC3** —— 新增静态守卫：`styles.css` 中任何焦点态规则若把指示画在 border box 外面，必须命中「固有尺寸控件」白名单；新增满宽控件的外扩环立刻红。
4. **AC4** —— 两个白名单都有**陈旧条目检测**：条目对应的违规已消失却没删条目，测试红（照搬 `a11y.spec.ts` 的 `KNOWN_VIOLATIONS` 模式）。
5. **AC5** —— 存量违规全部修复，基线白名单清空，审计转为硬失败。
6. **AC6** —— 四处历史容器补丁中，凡因根治而失效的，要么删除、要么在注释里写明「为哪一类**仍然外扩**的控件保留」，不留无主补丁。
7. **AC7** —— `bun run typecheck && bun run lint && bun run test && bun run format:check` 全绿；Playwright chromium 全绿；单二进制 build smoke 通过。

## 7. 备选方案与取舍

**A · 继续给容器补 padding（现状修法）** —— 拒绝。O(容器数) 且随时间增长，四次实践已证明必然从未覆盖的轴复发。

**B · 全量 inset：所有焦点环都画进控件内部** —— 结构上最彻底（集合 A 清空 ⇒ 交叉积恒为空 ⇒ 容器永远不用管）。但实心按钮需要重做焦点环（accent 环画在 accent 底上不可见，需 `inset` 双层：内圈底色细线 + 外圈 accent），全站按钮/页签/卡片焦点视觉可见变化。**用户已明确选择先上守卫、逐个修，未选此方案**；保留为后续可选演进——本 RFC 的守卫上线后，若哪天想切 B，守卫正好能证明切换无回归。

**C · 只做守卫不修存量** —— 拒绝。止血但不治病，白名单永久挂着等于默认接受缺陷。

**D · 只做静态 CSS 规则守卫** —— 拒绝作为唯一手段。静态只能测代理条件（容器有没有 padding），既有假阳（容器根本不含可聚焦子元素）又有假阴（padding 够但子元素负 margin 顶出去）。**真实几何只有真实布局引擎能判定。** 静态层保留，但只负责它擅长的那半边（见 design.md §2）。

## 8. 待用户确认

无。方向（先上守卫 → 基线白名单 → 逐条修 → 转硬失败）与守卫强度（静态 + Playwright 双层）已由用户于 2026-07-20 拍板。
