# RFC-209 工作组房间回合语义归位（自由协作不再假装有回合 + 回合号单一事实源）

状态：**Draft，待用户批准**（v2 —— 已折入对抗设计门 1 P0 + 3 P1 + 4 P2，见 design.md §9）
提出日期：2026-07-20
触发：用户实测反馈——「在工作组的自由讨论里，发现第 x 回合的轮次总是跳，并且中间一直穿插第 0 回合」

---

## 1. 背景

工作组聊天室（`WorkgroupRoom`）在消息流里按 `workgroup_messages.round` 画「第 X 回合」分隔线。
用户在**自由协作（free_collab）**房间里看到的实际效果是：

```
   （无分隔）×6         ← 首轮全员规划，全部 round=0
── 第 3 回合 ──
── 第 0 回合 ──         ← 人在房间里说了一句话
── 第 5 回合 ──
── 第 0 回合 ──         ← 又说了一句
── 第 8 回合 ──
── 第 11 回合 ──
```

回合号既**跳号**（3 → 5 → 8 → 11）又**回退**（穿插第 0 回合）。

调查结论：**这不是一个 bug，是三个独立缺陷叠加**，加上一个前端放大器。详细技术根因见
`design.md §1`，此处只给产品视角的一句话版本：

| # | 缺陷 | 影响模式 | 症状 |
|---|---|---|---|
| ① | fc 的「回合号」其实是**成员 run 累计计数**（`max_rounds` 预算表），不是回合序数 | free_collab | 跳号 |
| ② | 并发轮攥着**过期快照**写库，早期波次的消息永远写 0 | free_collab 为主 | 大片 round 0 |
| ③ | 所有 HTTP 路由写的人类 / 系统消息**硬编码 `round: 0`** | 两种模式 | 中间穿插第 0 回合 |
| 放大器 | 前端只要 round **变化**就画分隔线，**包括回退**（顺带产生重复 React key） | 两种模式 | 回退也画一条 |
| 附带 | `reviveKilledClarifyContinuations` 重铸后，被杀行与重铸行**同时**进账本 | **两种模式** | lw 跳 1；fc 一个逻辑轮吃掉 2 格预算 |

> **v2 更正（对抗门 P1-4）**：v1 说「lw 免疫、每轮 +1」是过度断言。lw 账本 = `max(wg_round)` + NULL 尾巴，
> 所以**既会跳**（引擎外 mint 的续跑行不带 `wgRound`，每个未打戳的 NULL 行立刻 +1）**也会退**
> （在飞 leader 行被 cancel 后从 `max` 与尾巴里一并排除）。lw 的回合号是真序数，但账本读数本身不单调
> ——这正是前端单调守卫必须无条件存在的理由。详见 design §1.7。

其中 ① 是设计层面的**类别错误**：`design/RFC-164-workgroup/design.md:262` 白纸黑字写明
fc 的硬顶是「**成员 run 总数** > max_rounds」——这个数从来就是预算表，不是回合序数。
自由协作模式下成员是**各自异步认领任务**的，本质上**不存在全局回合**；把预算计数器
标成「第 X 回合」是在给一个不存在的东西编号。

---

## 2. 目标

1. **自由协作房间不再显示回合分隔线**——因为它本来就没有全局回合。
2. **把 `max_rounds` 预算如实呈现**为右栏的「成员发言预算 已用 / 上限」，让用户能看到
   真正决定任务生死的那个数（触顶 = 任务 failed）。
3. **人类 / 系统消息归入「当前回合」**（lw）——房间发言、@ 派单、确认门决定、中途改配置不再
   硬编码 0。**fc 下该列恒 0**（D10：该模式无回合语义，写预算计数器等于把本 RFC 自己认定的
   类别错误存进库）。目标收敛为「**lw 下** `round` 列可信」——lw 账本本身不单调（§1 注），
   所以不承诺「处处单调」。
4. **Leader-Worker 房间的分隔线单调不回退**——回退与重复一律并入上文。
5. **卡片轮号改读权威列**：`leaderRoundOf` 从消息 round 反推的做法退役，改读 `node_runs.wg_round`。
   （不做这条，目标 3 会直接打断 RFC-182 的卡片锚定——见 design §1.8 / §2.5。）
6. 修掉被重铸的反问续跑双计——**lw 与 fc 都修**。

## 3. 非目标

- **不改 `max_rounds` 的预算语义**。fc 仍按成员 run 总数计、lw 仍按 leader 轮序数计；
  `deriveRoundsUsed` 的返回值与旧 `countRoundsUsed` 逐值相同（只是搬家 + 多几个调用方）。触顶行为、
  宽限收尾轮（RFC-187 §3-7）、`isLeaderWrapUpContinuation` 全部原样。
  **唯一例外** = AC-11 的「被取代的被杀反问续跑行」排除——那是**修正双计**，不是改口径。
- **不动 `node_runs.wg_round` 的三处打戳**（`:1269` 确认门 holder / `:1953`、`:1963` 被 @ 轮成员戳）。
  v1 只保护了第一处，机械清扫会误伤后两处（design §2.4 保护名单）。
- **不给 fc 编造波次序数**。用户已明确否决「每批唤醒 +1」方案。
- **不做 migration / 不回填存量数据**。存量 fc 房间因为不再画分隔线而自然干净；存量 lw
  房间的 round-0 人类消息由前端单调守卫吸收。
- **不改工作组表单里 `maxRounds` 字段的编辑体验**（表单里「最大轮数」在 fc 下同样名不副实，
  但那是另一处文案问题，留作 follow-up）。
- 不动 `node_runs.wg_round` 在 leader_worker 上的账本地位——那是唯一真正承重的数
  （审计确认：改它就是改 max_rounds 上限）。

## 4. 用户故事

- **US-1**（自由协作观察者）作为在自由协作房间里看进展的人，我看到的是一条连续的讨论流，
  没有跳号的假回合分隔；右栏告诉我「成员发言预算 7 / 20」，我知道还剩多少额度、什么时候
  会触顶。
- **US-2**（房间参与者）作为在房间里插话的人，我发的消息落在**当前正在进行的那一轮**下面，
  不会把消息流劈成「第 5 回合 / 第 0 回合 / 第 5 回合」三段。
- **US-3**（Leader-Worker 观察者）作为看 lw 房间的人，回合分隔线单调递增：第 1、第 2、第 3；
  任何迟到的、归属早期回合的消息安静地并入当前段落，不再倒画一条分隔线。
- **US-4**（运维）作为排障的人，`workgroup_messages.round` 列的值可以直接用来判断消息属于
  哪一轮，不用再区分「这条是引擎写的还是路由写的」。

## 5. 方案概要（详见 design.md）

1. **回合账本单一事实源**：把 `countRoundsUsed` 的推导抽成 `services/workgroupRounds.ts`
   的纯函数 + 一个轻量 DB 读取器，引擎与路由**共用同一个**。
2. **写入规则归一**：`workgroup_messages.round` = 「写入时刻账本读数」，仅两处例外
   （leader 轮自身产出用该轮 `wgRound`；派单卡的结果 / 失败 / 交付 / 取消用 `assignment.round`）。
   开场目标消息（前奏）恒为 0。
3. **前端分隔线单调**：只在 `round > 迄今最大值` 时画；回退与重复并入上文。
4. **fc 关闭分隔线**：房间按模式派生 `roomShowsRoundDividers(mode)`，fc 恒 false。
5. **fc 预算表**：房间聚合新增 `roundsUsed`（复用已加载的 host run 列，零新查询），右栏
   复用现有 `workgroup-room__info` 的 dt/dd 行渲染「已用 / 上限」。
6. **rider**：`reviveKilledClarifyContinuations` 给被杀行与重铸行打同一个 `wg_round` 戳，
   消除双计。

## 6. 验收标准

| # | 验收 | 覆盖 |
|---|---|---|
| AC-1 | free_collab 房间**不出现任何**「第 X 回合」分隔线（`wg-round-*` testid 一个都没有） | US-1 |
| AC-2 | free_collab 房间右栏显示「成员发言预算 N / M」，N 等于后端账本读数（与触顶判据同源） | US-1 |
| AC-3 | leader_worker 房间的分隔线序列**单调递增**；给定含回退 round 的消息流，回退处不画线 | US-3 |
| AC-4 | **lw**：房间发言 / @ 派单 / 确认门决定 / 中途改配置写入的消息，`round` 等于写入时刻账本读数，不是 0。**fc**：恒 0 | US-2、US-4 |
| AC-5 | 人 @ 派单产生的 assignment 行 `round` 同上；其结果消息随之落在正确回合下 | US-2 |
| AC-6 | `deriveRoundsUsed` 与改动前 `countRoundsUsed` 对同一批输入**逐值相同**（互 oracle 测试）；AC-11 的被取代行排除是唯一允许的差异 | 非目标 §3 |
| AC-7 | max_rounds 触顶行为、宽限收尾轮、fc 死锁判定全部不变（既有测试零改动通过） | 非目标 §3 |
| AC-8 | 派单卡族（结果 / 失败 / 交付 / 取消）的 `round` 仍恒等于 `assignment.round`——「轮 2 派、轮 7 收工」的结果不被标成 7、不与其派单卡脱钩 | D13 |
| AC-9 | 开场目标消息仍是 round 0 且**不产生**分隔线（前奏语义保持） | 回归 |
| AC-10 | leader 卡片轮号读 `node_runs.wg_round`（NULL 回退旧推导）；协议重试与其本轮**共享**轮号；design §1.8 的失败场景下卡片不再被冲到房间底部（改动前必红） | 目标 5 |
| AC-11 | 被重启杀死的反问续跑被重铸后，账本在 **lw 与 fc** 都不再多计（两条红→绿回归测试） | 目标 6 |

## 7. 决策记录

| # | 问题 | 决策 | 依据 |
|---|---|---|---|
| D1 | 用户看到问题的模式 | free_collab | 用户 2026-07-20 回答 |
| D2 | fc 的「第 X 回合」应该表示什么 | **不显示回合分隔**，改右栏预算表 | 用户拍板；fc 无全局回合，编造序数是类别错误 |
| D3 | 人类 / 系统消息的 round 归属 | **归入「当前回合」**（写入时读实时账本） | 用户拍板；语义最直白，且让 round 列处处可信 |
| D4 | 前端是否加单调守卫 | 加，无条件 | 纵深防御：任何写入侧遗漏都不会再表现为回退分隔线 |
| D5 | 是否回填存量数据 | 否，零 migration | fc 不再画线自然干净；lw 由 D4 吸收 |
| D6 | 是否给 fc 编造波次序数 | 否 | 用户在三选项中否决 |
| D7 | 预算表是否也在 lw 显示 | 后端字段两种模式都发（同一 oracle），**UI 只在 fc 渲染** | lw 的回合号已由分隔线传达；不给 lw 加新视觉元素 |
| D8 | 双计 rider 是否并入本 RFC | 并入，独立子任务 + 独立测试 | 同域、同一个「回合号跳」症状族；且 fc 侧会直接污染 AC-2 要上 UI 的那个数 |
| D9 | 【v2，P0 处置】`leaderRoundOf` 从消息 round 反推是否保留 | **退役**，改读 `node_runs.wg_round`，NULL 回退旧推导 | 账本返回的是**在飞轮**（`:1491` mint 早于 `:1512` run），目标 3 一落地反推序数必漂移、卡片被冲到房间底部 |
| D10 | 【v2，P2-9】fc 下 `round` 列写什么 | **恒 0**（消息与 assignment 两张表） | v1 让 fc 继续携带预算计数器，与本 RFC「那是类别错误」的论证自相矛盾；恒 0 即「此模式无回合」，且把 live read 收窄到 lw |
| D11 | 【v2，P1-2】rider 的手段 | 从「给两行打同一个 `wgRound` 戳」改为**派生层排除被取代行** | 打戳对 fc 是 no-op（fc 分支不读 `wgRound`）、会违反 RFC-189「fc 成员行恒 NULL」契约、且 NULL 尾巴 k≥2 时等价于不修 |
| D12 | 【v2，P1-5】路由层如何杜绝「省略即写 0」 | 共享行构造器 `buildRoomMessageRow`，**`round` 必填无默认** | 四处路由是裸 insert 不走 `postMessage`，而 schema 有 `.default(0)`；字面量表锁抓不到「省略」 |
| D13 | 【v2，攻击面 7】派单卡族是否也享受极性翻转 | **否**，独立入口 `postAssignmentMessage`，`round` 必填 | 这一族账本读数**定义上就是错答案**（轮 2 派、轮 7 收工的结果会被标 7 并与派单卡脱钩），不适用「省略即兜底」 |
