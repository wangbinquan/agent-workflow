# RFC-202 任务生命周期出口与终态清场（proposal）

- 状态：Draft（待用户批准）
- 来源：[全仓 UX/功能/错误呈现审计（2026-07-16）](../ux-functional-audit-2026-07-16.md) §2 F-0/F-12/F-13/F-14/F-15、§1 R4（resumeTask 伪成功面）与 R8（终态不清场）
- 关联：RFC-097（生命周期 CAS 与转移表）、RFC-122（反问指令）、RFC-158（评审导航）

## 1. 背景

审计确认了一组同主题的高严重度问题：**任务停进某个状态之后，用户失去出口，或者系统留下的"待办"永远不清场**。它们共享同一个产品语义缺陷——生命周期只设计了"往前走"的路，没有设计"退出/放弃/善后"的路：

1. **P0**：多文档评审上游产出空列表（审计 agent 零发现，本是成功态）时，任务永久卡死在 `awaiting_review`，且该评审在收件箱/详情/画布/API 四个入口全部不可见、不可决策，唯一出路是取消任务丢弃全部进度。
2. **P1**：`awaiting_human` / `awaiting_review` 的任务既不能取消也不能删除——想放弃的用户没有任何出口（服务门比共享转移表窄；`task-not-cancelable` 文案还谎称「该任务已结束」）。
3. **P1**：daemon 正常重启把在跑任务错标为「已取消 / canceled by user」，取消态无恢复入口；幸存者路径写的 `daemon-shutdown` 又与 autoResume 只认的 `daemon-restart` 对不上——重启后没有任何任务能被自动恢复。
4. **P1**：修复弹窗里 `resumeTask` 失败被吞（HTTP 200 + ok:false，前端当成功静默关窗）；反问/评审/下发三条路由的 resumeTask 同样 fire-and-forget，接口报成功、任务停在原地。
5. **P1**：删除工作流不检查定时任务引用，遗留的启用中定时任务此后每次到点静默失败、连挂 10 次才自动停用，全程不产生任何用户可见痕迹。
6. **R8**：已失败/已取消任务的反问轮永久滞留在收件箱与徽标里显示「待回答」（本机实测：收件箱 17 条反问几乎全部来自死任务，含等待 23 天的僵尸轮）；6 月初的评审轮至今挂着「待评审」；对死任务回答会先落库再弹未翻译的报错。

## 2. 目标

1. **消灭 P0**：空列表评审轮不再 park——自动通过并发布空 `accepted` 端口，任务照常续跑完成（零发现 = 成功，语义与 approveMultiDocReview 空子集通过一致）。
2. **给等待态一个出口**：`awaiting_human` / `awaiting_review` 可取消（对齐共享转移表），取消时封存该任务全部 open 反问/评审轮；修正 `task-not-cancelable` 文案。
3. **daemon 重启不再伪装成用户取消**：优雅关停中断的任务统一落 `interrupted` + `errorSummary='daemon-restart'`，与崩溃路径对齐，使「继续执行」按钮与 `autoResumeOnBoot` 都能覆盖；消除 `daemon-shutdown`/`daemon-restart` 的匹配裂缝。
4. **修复/续跑失败可见**：修复弹窗在 `ok:false` 时不关窗、展示失败原因与下一步；反问/评审/下发三条路由的 resumeTask 失败上浮为响应内可见字段并在前端呈现。
5. **删除工作流有引用保护**：存在启用中定时任务引用时 409 拦截并列出定时任务名（与 deleteAgent 的既有行为对称）。
6. **终态清场**：任务进入终态（done/failed/canceled）时封存其 open 反问/评审轮；收件箱、徽标、反问/评审列表的"待处理"口径全部排除终态任务；对已封存轮的回答/决策提交给出明确中文拒绝（而非先落库再报内部错误）。

## 3. 非目标

- 错误码 i18n 词典的全量补齐、ErrorBanner details 折叠渲染（审计路线 RFC-A，独立 RFC；本 RFC 只补自己新增/触及的错误码词条）。
- WS 断线重连补齐（RFC-F）、危险操作确认统一（RFC-E）、画布交互（RFC-D）。
- 任务的 DELETE（物理删除）能力——本 RFC 只补"取消"出口，不新增删除语义。
- `exhausted` 等失败态文案的全面翻译与恢复指引重写（RFC-A 范畴）。
- 多人并发竞态类问题（审计明确降权）。
- loop `port-count-lt` 校验（F-16）——属启动前置校验域，另行小 PR 处理，不并入本 RFC。

## 4. 用户故事

- **S1 空审计通过**：我用 Code→Audit→Fix 工作流跑了一单，审计 agent 零发现、输出空清单。任务不再卡在「等待评审」，而是自动通过该评审节点（详情页时间线可见"空清单自动通过"的记录），下游正常续跑直至完成。
- **S2 放弃反问**：agent 提了一堆我不想回答的问题，任务停在「等待人工」。我在任务详情页点「取消任务」（带确认），任务变为「已取消」，收件箱里它的反问同时消失；worktree 保留可手动取用。
- **S3 重启恢复**：我升级 daemon 重启后，之前在跑的任务显示「已中断」而不是「已取消」，详情页有「继续执行」按钮；如果我开了自动恢复，任务在 daemon 起来后自己续跑。
- **S4 修复失败看得见**：我对卡住的任务点「修复」，若状态改动生效但续跑失败，弹窗不再无声关闭，而是显示"已修复状态，但续跑失败：\<原因\>"并保留重试入口。
- **S5 删工作流被拦**：我删除一个还有启用中定时任务引用的工作流，系统 409 拦下并告诉我是哪几个定时任务在用，让我先处理它们。
- **S6 收件箱干净**：任务失败/取消后，它的反问与评审不再出现在收件箱、徽标计数与「待回答/待评审」列表里；我若通过历史链接打开旧反问页，页面明确显示"任务已结束，本轮已封存"且表单只读。

## 5. 验收标准

- A1 上游空列表的多文档评审节点：node_run 直接落 done、发布空 `accepted` 端口、任务可跑到终态；详情页/时间线有可见记录；不再出现"收件箱不可见的 awaiting_review"。回归锁：空列表 + 非空列表两条路径。
- A2 `cancelTask` allowedFrom 与共享转移表一致（含 awaiting_review/awaiting_human）；前端 awaiting 态显示取消按钮；取消后该任务 open 反问/评审轮全部封存；`task-not-cancelable` 文案更正为与事实一致。
- A3 优雅关停：30s 窗口内被中断的任务落 `interrupted`+`daemon-restart`（不再是 canceled）；超窗幸存者同样 `daemon-restart`（消除 daemon-shutdown 裂缝）；`autoResumeOnBoot` 开启时两类都被自动恢复；用户手动取消路径不受影响（仍为 canceled）。
- A4 修复弹窗：`ok:false` 时不关窗、渲染 outcomeMessage（中文映射）；反问/评审/下发三路由的 resume 失败在响应中可见且前端有呈现；三处均有回归测试。
- A5 deleteWorkflow：存在引用它的定时任务（enabled 或 disabled 均计）时 409 `workflow-scheduled-referenced`，details 带定时任务名清单；前端展示清单；无引用时行为不变。
- A6 清场：终态任务的 open 轮不再计入收件箱三源计数、不出现在待办列表；对已封存轮提交回答/决策 → 409 + 明确中文文案，答案不落库；存量历史数据（升级前已滞留的死轮）在升级后同样从待办中消失（口径过滤对历史行生效）。
- A7 全部改动带正向/边界/回归测试；`bun run typecheck && bun run test && bun run format:check` 与前端 vitest 全绿。
